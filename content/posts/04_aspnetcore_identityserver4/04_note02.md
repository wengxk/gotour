## 1. 介绍

OpenID Connect 是建立在 OAuth2.0 之上的身份验证层，它使得客户端能够通过授权服务器验证终端用户的身份标识 `id_token` ，能以REST风格的形式来获取用户的基本配置信息。并且 OpenID Connect 完全集成 OAuth2.0，即用户也可以获取 `access_token`，完成第三方授权的交互。简单的说，OpenID Connect 是在 OAuth2.0 框架之上扩展了`id_token`及用户相关信息的一个身份验证协议。

官网示意图如下：

![figure 1: OpenID Connect Protocol Suite](/images/04_aspnetcore_identityserver4/07_oidc.png)

## 1.1 相关术语

- EU: End User，终端用户
- OP: OpenID Provider，用户认证和授权的提供方，基本就是指授权服务器
- RP: Relying Party，一般就是指客户端或用户代理端

其他相关术语可以参见[官网](https://openid.net/specs/openid-connect-core-1_0.html#Terminology)。

## 1.2 基本工作流程

![figure 2: OpenID Basic Flow](/images/04_aspnetcore_identityserver4/08_oidc_basic_flow.png)

1. 客户端发送认证请求到授权服务器
2. 授权服务器请求用户的认证与授权过程
3. 授权服务器返回 `id_token` 和 `access_token`
4. 客户端发送请求到授权服务器的UserInfo端点，请求获取用户信息，并同发送 `access_token`
5. 授权服务器验证 `access_token` 通过后，返回用户信息

## 2. ID Token

ID Token 就是 OpenID Connect 协议对于 OAuth2.0 框架的一个最主要的扩展。 ID Token 是安全的用户令牌，表现为jwt形式，包含了用户的一些最基本的声明信息。
OpenID Connect 协议定义了以下最基本的一些用户声明：

- iss: Issuer Identifier，必须，签发方唯一标识
- sub: Subject Identifier，必须，用户唯一标识，最长255个ASCII字符
- aud: Audience(s)，必须，id token的受众标识，必须包含RP端的client_id
- exp: Expiration time，必须，id token的失效时间，超过该日期后，对应的id token必须不能被接受
- iat: Issued Time，必须，签发时间
- auth_time: End-User authentication time，1970-01-01 00:00:00起的UTC秒数，如果客户端发送的认证请求中包含max_age或者auth_time作为基本声明，则该参数在id token中必须
- nonce: 随机字符串，用来降低重放攻击或者可以包含其他session相关的信息。  
  如果客户端发送认证请求时包含nonce，则返回的id token也必须包含nonce，并且客户端在收到id token时必须检查该nonce和之前发送的nonce值是否一致
- acr: Authentication Context Class Reference，认证上下文
- amr: Authentication Methods References，认证的签名方法
- azp: Authorized party，如果又则必须包含client_id，一般只有在受众方和被认证的一方不一致时才使用

## 3. 认证

OpenID Connect 协议提供了三种认证流程：

- Authorization Code Flow：OAuth2.0的授权码模式
- Implicit Flow：OAuth2.0隐式
- Hybrid Flow ：Authorization Code Flow + Implicit Flow 混合模式

以上三种认证流程请求的参数 `response_type` 值范围如下：

|response_type 值|认证流程类型|
|:---|:---|
|code|Authorization Code Flow|
|id_token|Implicit Flow|
|id_token token|Implicit Flow|
|code id_token|Hybrid Flow|
|code token|Hybrid Flow|
|code id_token token|Hybrid Flow|

即对于授权码模式，在认证请求中参数 `response_type` 只能为code，若认证通过后，OP也只会返回code，如果RP需要获取id_token，则必须再请求到OP的token endpoint；对于Implicit模式，RP可以在认证请求通过后直接获取到id_token或者token，而不需要授权码；对于混合模式根据response_type的值OP会返回对应的信息。

|特征|Authorization Code Flow|Implicit Flow|Hybrid Flow|
|:---|:---|:---|:---|
|所有令牌都由Authorization Endpoint提供|否|是|否|
|所有令牌都由Token Endpoint提供|是|否|否|
|令牌不会暴露给用户代理|是|否|否|
|客户端可以认证|是|否|是|
|可能存在Refresh Token|是|否|是|
|所有通信一次性完成|否|是|否|
|多数通信为在服务器之间完成|是|否|不一定|

> 对于令牌不会暴露给用户代理这个特征我的理解是：对于授权码模式，RP请求认证通过后OP最多会以重定向的方式将code暴露给用户代理端，即用户可以在浏览器上看到code，但是RP在收到code之后的操作，比如获取id_token和其他的token则是在RP服务器端完成的，用户代理端即用户在浏览器是看不到这些token信息的。

### 3.1 基于Authorization Code Flow的认证请求与响应

#### 3.1.1 基本流程

![figure 3: OpenID With Authorization Code Flow](/images/04_aspnetcore_identityserver4/09_oidc_authorization_code_flow.png)

#### 3.1.2 请求参数

- scope，必须，必须包含值openid
- response_type，必须，此处值必为code
- client_id，必须，RP唯一标识
- redirect_uri，必须，重定向至RP的地址
- state：推荐，用户在OP和RP之间的不透明的状态值，可以减少跨站请求伪造
- 其他可选参数

#### 3.1.3 示例

RP向OP获取授权码的请求示例：

``` JavaScript
  GET /authorize?
    response_type=code
    &scope=openid%20profile%20email
    &client_id=s6BhdRkqt3
    &state=af0ifjsldkj
    &redirect_uri=https%3A%2F%2Fclient.example.org%2Fcb HTTP/1.1
  Host: server.example.com
```

OP返回给RP授权码的响应示例：

```JavaScript
HTTP/1.1 302 Found
  Location: https://client.example.org/cb?
    code=SplxlOBeZQQYbYS6WxSbIA
    &state=af0ifjsldkj
```

RP向OP的token endpoint获取token的请求示例：

```JavaScript
  POST /token HTTP/1.1
  Host: server.example.com
  Content-Type: application/x-www-form-urlencoded
  Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW

  grant_type=authorization_code&code=SplxlOBeZQQYbYS6WxSbIA
    &redirect_uri=https%3A%2F%2Fclient.example.org%2Fcb
```

> 此处的验证为http基本验证，形式为client_id:client_secret的形式

## 4. UserInfo Endpoint

用户基本配置信息在OAuth2.0中是一个受保护的资源，需要RP以token作为凭据获取。有时id_token无法满足RP的需求，则此时可以向UserInfo Endpoint获取用户配置。

```JavaScript
GET /userinfo HTTP/1.1
Host: server.example.com
Authorization: Bearer SlAV32hkKG
```

```JavaScript
  HTTP/1.1 200 OK
  Content-Type: application/json

  {
   "sub": "248289761001",
   "name": "Jane Doe",
   "given_name": "Jane",
   "family_name": "Doe",
   "preferred_username": "j.doe",
   "email": "janedoe@example.com",
   "picture": "http://example.com/janedoe/me.jpg"
  }
```

## 5. 总结

- OpenID Connect是建立在OAuth2.0之上的身份验证层协议
- 基本上OP和RP都是要建立在TLS之上的
- 关于 OpenID & ConnectOAuth2.0 的一些基本知识：jwt，http redirect， http authentication

在学些了OAuth2.0&OpenID Connect有助于我们理解互联网认证和授权的相关知识，也更加有助于使用相关的组件或平台应用。
